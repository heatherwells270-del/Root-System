# Root System — V3 Architecture
*Last updated: February 2026*

---

## Overview

Root System is a privacy-first mutual aid network for local communities. V3 is a React Native mobile application backed by a lightweight WebSocket relay server. Posts, time bank records, and community data are stored locally on each device in SQLite — the relay is a temporary buffer, not a database.

The guiding constraint: **no central authority holds your data.** The relay can be shut down tomorrow and your community's history survives intact on member devices.

## Cryptography Model

### Identity
- **Ed25519 keypair** generated on first launch using `expo-crypto` (OS-level RNG, 32 bytes)
- Private key stored in **SecureStore** — hardware-backed on supported devices, never serialized into JS state
- Public key is the user's permanent identifier on the network
- Device ID is a separate 16-byte random UUID used for vector clocks and rate limiting

### Post Signing
- Every post is signed before storage: `sign(canonicalPost(post), privateKey)` → Ed25519 signature
- Signature verified before any incoming post is persisted: unsigned or tampered posts are dropped
- Canonical form is deterministic — same input always produces the same string to sign

### Relay Buffer Encryption
- Community key: **AES-256-GCM** symmetric key, generated by the planter on community creation
- All posts pushed to the relay buffer are encrypted with the community key
- The community key itself is distributed to new members encrypted with their **X25519 public key** (converted from Ed25519 using `ed25519.utils.toMontgomery()`)
- The relay never sees plaintext post content — it stores and forwards encrypted blobs only

### Nonce / Challenge-Response Auth
- On WebSocket connect, relay sends a 32-byte cryptographically random nonce (`globalThis.crypto.getRandomValues`)
- Client signs `auth:{publicKey}:{deviceId}:{timestamp}` + `nonce:{nonce}` with its private key
- Relay verifies both signatures; hello timestamps must be within 60 seconds (replay protection)

## Data Model

All types defined in `src/models/types.ts`. Every stored record is signed.

| Type | Description | Sync |
|------|-------------|------|
| `Identity` | Local user — keypair, handle, bio, location | Local only |
| `Community` | Name, geography, planter key, zones, anchor devices | Relay buffer |
| `Post` | Offer / Need / Free. CRDT with version + updated_at | Relay buffer |
| `Exchange` | Time bank transaction. Dual-confirm required | Relay buffer |
| `Appeal` | Moderation challenge on a removed post | Relay buffer |
| `Coalition` | Sub-group within a community | Relay buffer *(stub)* |
| `KnowledgeEntry` | Community-written guide | Relay buffer *(stub)* |
| `TrustScore` | Computed from exchange history + flag record | Local only |
| `SyncItem` | Union type — the unit transmitted over the relay | — |

### CRDT Conflict Resolution
Posts use a version-based last-write-wins CRDT:
- On conflict: update only if `incoming.version > stored.version AND incoming.updated_at >= stored.updated_at`
- Tombstones are permanent — a deleted post can never be "un-deleted" by a peer sending an older version
- `flagPost()` uses a single atomic SQL statement with `json_each()` to prevent double-flag race conditions

## Sync Architecture

```
App foreground / community switch
    └── startSync(communityId)
          ├── RelayClient connects → challenge-response auth
          ├── join(communityId)
          ├── pullBuffer(communityId, since)  ← gets encrypted blobs since last pull
          ├── requestCommunityKey(communityId)  ← if no key stored yet
          └── flush _pendingPushes  ← posts saved while offline (capped at 100)

Incoming buffer-items
    └── _processBufferItems()
          ├── decrypt AES-256-GCM with community key
          ├── JSON.parse → Post
          ├── verify Ed25519 signature → drop if invalid
          ├── upsertPost() → SQLite CRDT merge
          ├── ackBufferItem() → relay deletes item
          └── advance sync watermark

Post created by user
    └── pushPostToRelay(post)
          ├── if not authed → push to _pendingPushes (drop oldest if full)
          ├── get community key from SecureStore
          ├── encryptForBuffer(JSON.stringify(post), communityKey)
          └── relay.pushToBuffer(communityId, encryptedBlob)
```

Phase 4 (planned): WebRTC device-to-device sync as a complement to relay sync. The relay is the primary path; WebRTC is a fallback for offline communities. Hook point is marked in `sync/index.ts`.

## Relay Architecture

Single-file Node.js WebSocket server. Stateless per-session — all state is in-memory (no database).

| File | Purpose |
|------|---------|
| `src/server.ts` | WebSocket server, message routing, rate limiting, key brokering |
| `src/store.ts` | In-memory: buffer items, connected clients, nonce store, key requests |
| `src/auth.ts` | Ed25519 verification, nonce generation, timestamp validation |

The relay:
- **Does not** store plaintext posts — buffer items are opaque encrypted blobs
- **Does not** know who posted what — it knows public keys and community IDs, not content
- **Does** broker community key requests between planters and new members
- **Does** rate limit: per-connection, per-community, per-buffer-push
- Requires Node.js **22.5+** — uses `node:sqlite` (added in Node 22.5) and `globalThis.crypto.getRandomValues` (no polyfill)

## Security Decisions & Trade-offs

| Decision | Rationale |
|----------|-----------|
| No accounts | Eliminates credential database, reduces COPPA and GDPR surface, enables anyone to participate without email |
| Ed25519 over RSA | Smaller keys, faster operations, pure-JS auditable implementation (`@noble/ed25519`) |
| AES-256-GCM for relay buffer | Standard symmetric encryption; community key distribution is the harder problem (solved with X25519 ECDH) |
| Relay sees only encrypted blobs | Limits relay operator liability; relay compromise does not expose post content |
| No internal messaging | Eliminates the primary channel for grooming, exploitation, and harassment |
| Approximate location only | Neighborhood precision is sufficient for local aid; exact coordinates are unnecessary and high-risk |
| CRDT with tombstone permanence | Deleted content stays deleted regardless of what peers broadcast |
| Relay is ephemeral | If relay goes down, local SQLite history is intact; relay is a sync facilitator, not the source of truth |

## Running Locally

### App
```bash
cd root-system-app
npm install
npx expo start
# Scan QR code in Expo Go (Android/iOS)
```

### Relay
```bash
cd root-system-relay
npm install
npm run dev
# Runs on ws://localhost:8080 by default
```

Set `EXPO_PUBLIC_RELAY_URL=ws://localhost:8080` in `root-system-app/.env.local`.

### Dependencies
- Node.js 22.5+ (relay)
- Expo SDK (app)
- Expo Go app on device for development
